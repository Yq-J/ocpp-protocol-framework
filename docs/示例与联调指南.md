# OCPP Protocol Framework 示例与联调指南

本文档补充可直接复制执行的本地联调步骤，帮助业务团队验证 WebSocket 接入、充电桩上报、平台下发命令和常见异常场景。示例模块位于 `ocpp-demo-business`，默认监听 `8080` 端口。

## 1. 启动前检查

在仓库根目录执行：

```bash
mvn clean test
mvn -pl ocpp-demo-business spring-boot:run
```

如果公司网络禁止访问 Maven Central，请先配置内网 Maven 镜像或预热本地仓库。当前工程依赖 Spring Boot、Spring WebSocket、Jackson、Lombok、JUnit 5 与 json-schema-validator。

## 2. 示例配置说明

`ocpp-demo-business/src/main/resources/application.yml` 已配置：

```yaml
server:
  port: 8080

ocpp:
  path: /ocpp/{chargePointId}
  connection-timeout-seconds: 60
  allow-unknown-actions: false
  max-text-message-bytes: 262144
  allowed-origins:
    - "http://localhost:8080"
```

本地联调时可以保持上述配置。生产环境不要照搬示例配置，至少应开启握手 Token、收敛 Origin、关闭默认示例 Handler，并按 `docs/生产化使用指南.md` 完成上线检查。

## 3. WebSocket 连接方式

充电桩或模拟器连接地址：

```text
ws://localhost:8080/ocpp/CP001
```

握手必须携带 OCPP 子协议之一：

| 协议版本 | WebSocket 子协议 |
| --- | --- |
| OCPP 1.6J | `ocpp1.6` |
| OCPP 2.0.1 | `ocpp2.0.1` |

如果使用 `websocat`，可以这样连接 OCPP 1.6：

```bash
websocat -H='Sec-WebSocket-Protocol: ocpp1.6' ws://localhost:8080/ocpp/CP001
```

如果使用 Postman、Apifox 或浏览器 WebSocket 客户端，请在请求 Header 或客户端设置中加入 `Sec-WebSocket-Protocol: ocpp1.6`。

## 4. 充电桩上报示例

### 4.1 OCPP 1.6 BootNotification

发送：

```json
[2,"boot-001","BootNotification",{"chargePointVendor":"DemoVendor","chargePointModel":"DemoModel"}]
```

预期返回类似：

```json
[3,"boot-001",{"currentTime":"2026-06-22T00:00:00Z","interval":300,"status":"Accepted"}]
```

### 4.2 OCPP 1.6 Authorize

发送：

```json
[2,"auth-001","Authorize",{"idTag":"USER001"}]
```

预期返回：

```json
[3,"auth-001",{"idTagInfo":{"status":"Accepted"}}]
```

### 4.3 OCPP 1.6 StatusNotification

发送：

```json
[2,"status-001","StatusNotification",{"connectorId":1,"errorCode":"NoError","status":"Available"}]
```

预期返回空 payload：

```json
[3,"status-001",{}]
```

### 4.4 OCPP 2.0.1 TransactionEvent

使用 `ocpp2.0.1` 子协议连接后发送：

```json
[2,"tx-001","TransactionEvent",{"eventType":"Started","timestamp":"2026-06-22T00:00:00Z","triggerReason":"Authorized","seqNo":1,"transactionInfo":{"transactionId":"TX001"}}]
```

预期返回：

```json
[3,"tx-001",{}]
```

## 5. 平台主动下发命令示例

平台下发接口要求目标充电桩已经通过 WebSocket 在线。先连接 `ws://localhost:8080/ocpp/CP001`，再调用 REST API。

### 5.1 连通性提示

```bash
curl -s http://localhost:8080/api/charge-points/CP001/ping
```

### 5.2 OCPP 1.6 RemoteStartTransaction

```bash
curl -s -X POST http://localhost:8080/api/charge-points/CP001/ocpp16/remote-start \
  -H 'Content-Type: application/json' \
  -d '{"connectorId":1,"idTag":"USER001"}'
```

此时充电桩模拟器应收到平台下发的 CALL，例如：

```json
[2,"<framework-generated-unique-id>","RemoteStartTransaction",{"connectorId":1,"idTag":"USER001"}]
```

模拟器需要在 `connection-timeout-seconds` 过期前返回同一个 uniqueId 的 CALLRESULT：

```json
[3,"<framework-generated-unique-id>",{"status":"Accepted"}]
```

### 5.3 OCPP 1.6 RemoteStopTransaction

```bash
curl -s -X POST http://localhost:8080/api/charge-points/CP001/ocpp16/remote-stop \
  -H 'Content-Type: application/json' \
  -d '{"transactionId":200001}'
```

模拟器返回：

```json
[3,"<framework-generated-unique-id>",{"status":"Accepted"}]
```

## 6. 常见异常与定位

| 现象 | 常见原因 | 建议处理 |
| --- | --- | --- |
| 握手失败或连接立即断开 | 未携带 `Sec-WebSocket-Protocol`、子协议不在白名单、Origin 不匹配 | 检查客户端子协议和 `ocpp.supported-sub-protocols`、`ocpp.allowed-origins`。 |
| 收到 CALLERROR `NotImplemented` 或 `ProtocolError` | Action 未在元数据中登记或没有业务 Handler | 确认 Action 名大小写、协议版本和 `@OcppActionMapping` 是否匹配。 |
| 收到 Schema 校验错误 | Payload 缺少必填字段或字段类型不符合 JSON Schema | 对照 OCPP 官方报文结构修正模拟器 payload。 |
| REST 下发命令超时 | 充电桩未在线、返回 uniqueId 不一致、未返回 CALLRESULT/CALLERROR | 先调用 `/ping`，再检查 WebSocket 侧是否收到 CALL 并按原 uniqueId 回复。 |
| 大报文被拒绝 | 超过 `ocpp.max-text-message-bytes` | 结合真实 MeterValues / 日志报文大小调整配置，并在网关层同步配置限制。 |

## 7. 继续接入业务系统的最小步骤

1. 业务工程引入 `ocpp-spring-boot-starter`。
2. 根据环境配置 WebSocket 路径、子协议、Origin、Token 和报文大小限制。
3. 为每个需要处理的桩侧 Action 编写 `@OcppActionMapping` 方法。
4. 注入 `OcppGateway`，使用便捷方法或通用 `call` 下发平台命令。
5. 替换或扩展 `OcppSessionRepository`，解决多实例部署的在线会话与命令路由。
6. 为交易、计费、设备状态、证书、固件和智能充电建立业务状态机，不能只依赖 DTO 与 Schema 校验。
