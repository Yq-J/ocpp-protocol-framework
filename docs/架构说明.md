# OCPP 协议框架架构说明

作者：JYq

## 1. 总体分层

```text
业务系统
  -> ocpp-spring-boot-starter
      -> WebSocket 接入
      -> 注解处理器注册
      -> 主动下发模板
      -> Redis 会话注册
      -> Redis Pub/Sub 跨节点转发
      -> 高负载保护
  -> ocpp-core
      -> 协议帧模型
      -> 编解码器
      -> DTO
      -> Handler 抽象
      -> Session 抽象
```

模块职责：

1. `ocpp-core`：纯协议核心，不依赖 Spring，定义 OCPP-J 帧、编解码、错误、DTO、会话和处理器抽象。
2. `ocpp-spring-boot-starter`：Spring Boot 接入层，负责自动配置、WebSocket 传输、业务方法注册、主动下发、Redis 会话注册和跨节点转发。
3. `ocpp-demo-business`：业务示例层，演示如何通过 starter 接入业务处理器和平台下发接口。

## 2. WebSocket 接入模型

框架使用原生 Spring WebSocket 文本帧，不使用 STOMP、SockJS 或 Netty。

```text
充电桩
 -> ws://host:port/ocpp/{chargePointId}
 -> OcppHandshakeInterceptor 提取 chargePointId
 -> OcppWebSocketHandshakeHandler 选择 ocpp1.6 / ocpp2.0.1 子协议
 -> OcppWebSocketHandler.afterConnectionEstablished
 -> sessionRepository.save(SpringOcppConnection)
```

`SpringOcppConnection` 封装 Spring `WebSocketSession`，向 core 层暴露统一的 `OcppConnection` 接口。这样 core 层不依赖 Spring WebSocket。

## 3. 充电桩上报调用链

```text
WebSocket 文本帧
 -> OcppWebSocketHandler.handleTextMessage
 -> OcppHighLoadGuard.allow
 -> 入站线程池
 -> OcppCodec.decode
 -> OcppCall
 -> OcppSchemaValidator.validate
 -> OcppHandlerRegistry.get(version, action)
 -> OcppActionHandler.handle
 -> @OcppActionMapping 业务方法
 -> OcppCodec.encodeCallResult / encodeCallError
 -> WebSocket 文本帧
```

入站处理线程池用于将 JSON 解析、校验、业务调用和响应发送从 WebSocket I/O 线程中卸载出来。队列满或限流触发时，框架会快速返回错误，避免继续堆积任务。

## 4. 平台下发调用链

### 4.1 单节点或本地命中

```text
业务代码
 -> OcppGateway.call / OcppTemplate.call
 -> sessionRepository.get(chargePointId)
 -> OcppTemplate.callLocal
 -> 生成 OCPP uniqueId
 -> pendingRequests.put(uniqueId, future)
 -> OcppCodec.encodeCall
 -> OcppConnection.send
 -> 充电桩
 -> CALLRESULT/CALLERROR
 -> OcppWebSocketHandler.handleTextMessageAsync
 -> OcppTemplate.completeResult / completeError
 -> CompletableFuture 完成
```

`pendingRequests` 是节点内内存结构，用 OCPP `uniqueId` 关联平台下发请求和充电桩响应。

### 4.2 多节点本地未命中

当请求到达节点 A，但充电桩连接在节点 B：

```text
节点 A OcppTemplate.call
 -> 本地 sessionRepository.get(chargePointId) 未命中
 -> RedisBackedOcppSessionRepository.lookupNodeId(chargePointId)
 -> Redis GET ocpp:session:cp:{chargePointId}
 -> 得到 targetNodeId = 节点 B
 -> RedisOcppClusterForwarder.forward
 -> Redis PUBLISH ocpp:cluster:node:{节点 B}

节点 B RedisOcppClusterForwarder.onMessage
 -> handleRequest
 -> localDispatcher.dispatch
 -> OcppTemplate.callLocal
 -> 本地 WebSocket 发送给充电桩
 -> 收到 CALLRESULT/CALLERROR 后完成本地 Future
 -> publishResponse 到 ocpp:cluster:node:{节点 A}

节点 A RedisOcppClusterForwarder.onMessage
 -> handleResponse
 -> responseWaiters.remove(requestId)
 -> 完成最初业务侧 Future
```

这里有两个不同的关联 ID：

- OCPP `uniqueId`：节点 B 本地发送给充电桩的 OCPP CALL ID，用于关联桩响应。
- 跨节点 `requestId`：节点 A 和节点 B 之间 Redis Pub/Sub 消息的请求响应 ID，用于关联跨节点响应。

## 5. 会话存储架构

### 5.1 单机内存仓储

`InMemoryOcppSessionRepository` 保存：

```text
chargePointId -> OcppConnection
sessionId     -> chargePointId
```

适用于本地开发、单节点部署或测试。

### 5.2 Redis 会话注册仓储

`RedisBackedOcppSessionRepository` 同时维护本地连接和 Redis 归属索引：

```text
本地 JVM:
  chargePointId -> OcppConnection
  sessionId     -> chargePointId

Redis:
  ocpp:session:cp:{chargePointId} -> nodeId
```

本地 JVM 的连接表用于当前节点低延迟直发。Redis 归属索引用于其它节点判断应该转发到哪个节点。

断开连接时只删除仍归属于当前节点的 Redis key，避免旧连接断开误删新连接路由。

## 6. Redis Pub/Sub 转发架构

每个节点订阅一个私有 channel：

```text
ocpp:cluster:node:{nodeId}
```

`nodeId` 和 channel 是一一对应关系。其它节点只需要知道目标 `nodeId`，就能构造目标 channel。

request 消息表示“请目标节点代为向本地连接的充电桩下发命令”：

```json
{
  "kind": "request",
  "requestId": "uuid",
  "sourceNodeId": "node-a",
  "chargePointId": "CP001",
  "version": "OCPP_16",
  "action": "RemoteStartTransaction",
  "payload": {}
}
```

response 消息表示“目标节点已经执行完成，结果返回来源节点”：

```json
{
  "kind": "response",
  "requestId": "uuid",
  "ok": true,
  "payload": {}
}
```

Redis Pub/Sub 不保证离线消息投递，因此跨节点转发依赖目标节点在线且订阅正常。

## 7. 高负载保护

`OcppHighLoadGuard` 在 `OcppWebSocketHandler.handleTextMessage` 入口执行。

支持两种模式：

1. 单机本地计数：`max-inbound-messages-per-second > 0` 且未启用 Redis 全局限流。
2. Redis 全局计数：`redis-global-guard-enabled=true` 且存在 `StringRedisTemplate`。

Redis 全局限流 key：

```text
ocpp:inbound:qps:{epochSecond}
```

超过阈值时，WebSocket 层直接返回错误，不进入入站业务线程池。

## 8. 自动配置

`OcppAutoConfiguration` 负责装配：

| Bean | 说明 |
| --- | --- |
| `OcppCodec` | 默认 `DefaultOcppCodec`。 |
| `OcppSchemaValidator` | 默认 `NoopOcppSchemaValidator`。 |
| `OcppSessionRepository` | Redis 开启且 RedisTemplate 存在时使用 `RedisBackedOcppSessionRepository`，否则使用 `InMemoryOcppSessionRepository`。 |
| `RedisMessageListenerContainer` | RedisConnectionFactory 存在时创建，用于 Pub/Sub 订阅。 |
| `RedisOcppClusterForwarder` | `cross-node-forward-enabled=true` 且 Redis 组件存在时创建。 |
| `OcppTemplate` | 平台主动下发入口，并向 forwarder 注入本地 dispatcher。 |
| `OcppWebSocketHandler` | WebSocket 主处理器。 |
| `OcppWebSocketConfigurer` | 注册 WebSocket endpoint。 |
| `OcppAnnotatedHandlerRegistrar` | 扫描 `@OcppActionMapping` 业务方法并注册到 `OcppHandlerRegistry`。 |

## 9. 扩展点

| 扩展点 | 用途 |
| --- | --- |
| `OcppVersion` | 新增协议版本和 WebSocket 子协议映射。 |
| `OcppCodec` | 替换或增强 OCPP-J 帧编解码。 |
| `OcppActionHandler` | 直接注册 Action 处理器。 |
| `@OcppActionMapping` | 业务方法级处理器注册。 |
| `OcppSchemaValidator` | 接入官方 JSON Schema 校验。 |
| `OcppSessionRepository` | 替换在线会话存储，实现自定义集群路由。 |
| `OcppGateway` / `OcppTemplate` | 平台向桩下发命令。 |
| `RedisOcppClusterForwarder` | 当前 Redis Pub/Sub 转发实现，可替换为 MQ、RPC 或网关路由。 |

## 10. 架构边界

- WebSocketSession 不跨节点共享。
- Redis 只保存归属索引、限流计数和 Pub/Sub 消息，不保存连接对象。
- PendingRequest 是节点内内存结构，节点重启后未完成请求会丢失。
- Redis Pub/Sub 不持久化消息，目标节点离线时跨节点转发会失败或超时。
- 内置 DTO 覆盖常见动作，不等于覆盖全部 OCPP 字段和全部 Action。
- 默认 `NoopOcppSchemaValidator` 不做强 Schema 校验，生产认证场景应接入官方 JSON Schema。
