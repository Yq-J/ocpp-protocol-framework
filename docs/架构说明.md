# OCPP 协议框架架构说明

作者：JYq

## 分层

1. `ocpp-core`：纯协议核心，不依赖 Spring。
2. `ocpp-spring-boot-starter`：Spring Boot 接入层，负责自动配置和 WebSocket 适配。
3. `ocpp-demo-business`：业务示例层，只依赖 starter。

## 充电桩上报调用链

```text
WebSocket 文本
 -> OcppWebSocketHandler
 -> OcppCodec.decode
 -> OcppHandlerRegistry
 -> @OcppActionMapping 业务方法
 -> OcppCodec.encodeCallResult / encodeCallError
 -> WebSocket 文本
```

## 平台下发调用链

```text
业务代码
 -> OcppGateway 便捷方法 / OcppTemplate.call
 -> OcppSchemaValidator.validate
 -> OcppCodec.encodeCall
 -> OcppConnection.send
 -> 充电桩
 -> CALLRESULT/CALLERROR
 -> OcppTemplate.completeResult / completeError
 -> CompletableFuture 完成
```

## 扩展点

| 扩展点 | 用途 |
| --- | --- |
| `OcppVersion` | 新增协议版本和子协议映射。 |
| `OcppCodec` | 替换或增强帧编解码。 |
| `OcppActionHandler` | 直接注册 Action 处理器。 |
| `@OcppActionMapping` | 业务方法级处理器注册。 |
| `OcppSchemaValidator` | 替换或增强默认官方 JSON Schema 校验。 |
| `OcppSessionRepository` | 替换在线会话存储，实现集群适配。 |
| `OcppGateway` / `OcppTemplate` | 平台向桩下发命令，包含常用动作便捷方法和通用 call。 |
| `OcppProperties.allowedOrigins` | 收敛 WebSocket Origin 白名单。 |
| `OcppProperties.requireAuthToken` | 启用握手 Token 校验。 |
| `OcppProperties.maxTextMessageBytes` | 限制单条 WebSocket 文本帧大小。 |

## 生产化协议覆盖基线

框架现在把“协议动作覆盖”“Schema 校验”和“业务处理实现”拆分：

1. `OcppActionMetadata` 维护 OCPP 1.6J 与 OCPP 2.0.1 的完整 Action 集合，并通过 `OcppActionDescriptor` 暴露请求 DTO、响应 DTO、请求 Schema 路径和响应 Schema 路径；所有 Action 均映射到独立 Request/Response 模型类，不再以 `JsonNode` 作为元数据 DTO 类型。
2. `OfficialOcppSchemaValidator` 是 starter 默认的 `OcppSchemaValidator` 实现，会在业务 Handler 执行前完成协议版本、Action、Payload 对象形态和 JSON Schema 约束校验。
3. Schema 文件随 starter 打包，目录为 `schemas/ocpp1.6` 和 `schemas/ocpp2.0.1`，命名规则为 `{Action}Request.json` 与 `{Action}Response.json`；元数据中的路径是校验器、文档和后续代码生成的统一入口。
4. 默认业务处理器是示例/兜底能力，默认不注册；只有设置 `ocpp.enable-default-handlers=true` 时才启用。生产项目应使用 `@OcppActionMapping` 或自定义 `OcppActionHandler` 覆盖真实业务动作。
5. 如果企业需要替换为经过内审或扩展的 Schema 包，可以声明自己的 `OcppSchemaValidator` Bean；自动配置上的 `@ConditionalOnMissingBean` 会让自定义实现优先。

该设计避免把业务语义写入协议层，同时为全量 Action、Schema 校验、DTO 补齐、代码生成和版本升级提供统一元数据入口。

## 生产安全边界

框架内置的握手 Token、Origin 白名单和报文大小限制是基础防护能力。生产环境仍建议在负载均衡或 API 网关层启用 WSS、证书、IP 白名单、限流和访问审计；多节点部署必须替换默认单机内存会话仓储，并通过业务网关把平台命令路由到持有 WebSocket 连接的节点。
