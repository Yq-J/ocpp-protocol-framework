# OCPP 协议框架架构说明

作者：JYq

## 分层

1. `ocpp-core`：纯协议核心，不依赖 Spring。
2. `ocpp-spring-boot-starter`：Spring Boot 接入层，负责自动配置和 WebSocket 适配。
3. `ocpp-demo-business`：业务示例层，只依赖 starter。

## 充电桩上报调用链

```text
WebSocket 文本
 -> OcppWebSocketHandler
 -> OcppCodec.decode
 -> OcppHandlerRegistry
 -> @OcppActionMapping 业务方法
 -> OcppCodec.encodeCallResult / encodeCallError
 -> WebSocket 文本
```

## 平台下发调用链

```text
业务代码
 -> OcppTemplate.call
 -> OcppCodec.encodeCall
 -> OcppConnection.send
 -> 充电桩
 -> CALLRESULT/CALLERROR
 -> OcppTemplate.completeResult / completeError
 -> CompletableFuture 完成
```

## 扩展点

| 扩展点 | 用途 |
| --- | --- |
| `OcppVersion` | 新增协议版本和子协议映射。 |
| `OcppCodec` | 替换或增强帧编解码。 |
| `OcppActionHandler` | 直接注册 Action 处理器。 |
| `@OcppActionMapping` | 业务方法级处理器注册。 |
| `OcppSchemaValidator` | 接入官方 JSON Schema 校验。 |
| `OcppSessionRepository` | 替换在线会话存储，实现集群适配。 |
| `OcppGateway` / `OcppTemplate` | 平台向桩下发命令。 |

## 生产化协议覆盖基线

框架现在把“协议动作覆盖”和“业务处理实现”拆分：

1. `OcppActionMetadata` 维护 OCPP 1.6J 与 OCPP 2.0.1 官方 JSON Schema 暴露的完整 Action 名称集合。
2. `MetadataOcppSchemaValidator` 默认校验协议版本、Action 合法性和 Payload 对象形态，防止未定义 Action 进入业务处理。
3. 完整字段级校验仍通过 `OcppSchemaValidator` 扩展点接入官方 JSON Schema。生产环境建议在该扩展点中启用 required、enum、长度、格式和嵌套结构校验。
4. 默认业务处理器只保证框架可运行，不等同于生产业务实现；生产项目应使用 `@OcppActionMapping` 或自定义 `OcppActionHandler` 覆盖真实业务动作。

该设计避免把业务语义写入协议层，同时为全量 Action、Schema 校验、代码生成和版本升级提供统一元数据入口。
