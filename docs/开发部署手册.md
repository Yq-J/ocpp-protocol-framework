# OCPP 协议框架开发部署手册

## 1. 部署目标

本框架部署后提供原生 WebSocket OCPP 接入能力，并支持平台侧主动向充电桩下发命令。单节点部署时，连接、会话索引和 PendingRequest 都在同一个 JVM 内完成。多节点部署时，框架通过 Redis 维护 `chargePointId -> nodeId` 全局归属索引，并通过 Redis Pub/Sub 在节点之间转发下行命令。

核心原则：

- WebSocket 连接不能跨 JVM 共享，真正的连接对象只保存在接入该充电桩的节点本地内存中。
- Redis 只保存会话归属和限流计数，不保存 WebSocketSession。
- 跨节点下发时，来源节点先查 Redis 得到目标 `nodeId`，再 publish 到目标节点私有 channel。
- 目标节点收到消息后使用本地 WebSocket 连接发送 OCPP CALL，并将执行结果 publish 回来源节点。

## 2. 运行环境

| 项目 | 要求 |
| --- | --- |
| JDK | 1.8 |
| Spring Boot | 2.7.x |
| 构建工具 | Maven |
| WebSocket | Spring WebSocket 原生文本帧 |
| Redis | 单节点、哨兵或集群均可，生产建议高可用部署 |

starter 依赖 `spring-data-redis`，业务服务需要提供可用的 `StringRedisTemplate` 和 `RedisConnectionFactory`，通常通过 Spring Boot Redis 自动配置完成。

## 3. 单节点部署

单节点或本地开发可以不启用 Redis 会话注册和跨节点转发：

```yaml
server:
  port: 8080

ocpp:
  path: /ocpp/{chargePointId}
  connection-timeout-seconds: 60
  allow-unknown-actions: false
  inbound-worker-threads: 16
  inbound-queue-capacity: 200000
  max-inbound-messages-per-second: 0
  redis-session-registry-enabled: false
  cross-node-forward-enabled: false
  redis-global-guard-enabled: false
```

默认 WebSocket 地址：

```text
ws://{host}:{port}/ocpp/{chargePointId}
```

充电桩握手时必须携带 `Sec-WebSocket-Protocol`：

- OCPP 1.6：`ocpp1.6`
- OCPP 2.0.1：`ocpp2.0.1`

## 4. 多节点部署配置

多节点部署需要 Redis 配置，并建议同时开启 Redis 会话注册、跨节点转发和 Redis 全局限流：

```yaml
server:
  port: 8080

spring:
  redis:
    host: 127.0.0.1
    port: 6379
    timeout: 2000ms

ocpp:
  path: /ocpp/{chargePointId}
  connection-timeout-seconds: 60
  allow-unknown-actions: false

  inbound-worker-threads: 32
  inbound-queue-capacity: 300000
  pending-cleanup-interval-millis: 1000
  high-load-log-sampling-enabled: true

  max-inbound-messages-per-second: 12000
  redis-global-guard-enabled: true

  redis-session-registry-enabled: true
  cross-node-forward-enabled: true
  redis-session-registry-ttl-seconds: 180
  node-id: ${HOSTNAME:ocpp-node-01}
```

必须注意：

- `node-id` 必须全局唯一。容器环境建议使用 Pod 名、容器实例 ID 或主机名。
- `cross-node-forward-enabled=true` 只负责启用 Redis Pub/Sub 转发器。
- `redis-session-registry-enabled=true` 负责启用 Redis 会话归属索引。
- 实际跨节点下发要求两个开关同时启用，且 Redis Bean 可用。
- 如果只启用 `redis-session-registry-enabled`，框架只会写入归属索引，不会跨节点转发命令。
- 如果只启用 `cross-node-forward-enabled`，但没有 Redis 会话归属索引，来源节点不知道应该转发到哪个目标节点。

## 5. Redis Key 与 Channel

### 5.1 会话归属 Key

充电桩连接建立后，`RedisBackedOcppSessionRepository.save()` 写入：

```text
key   = ocpp:session:cp:{chargePointId}
value = {nodeId}
ttl   = ocpp.redis-session-registry-ttl-seconds
```

示例：

```text
ocpp:session:cp:CP001 -> ocpp-node-02
```

断开连接时，`remove()` 会先确认 Redis 当前值仍等于本节点 `nodeId`，再删除 key，避免旧连接断开时误删新节点写入的归属关系。

### 5.2 节点私有 Channel

每个节点启动时，`RedisOcppClusterForwarder` 订阅自己的私有 channel：

```text
ocpp:cluster:node:{nodeId}
```

示例：

```text
ocpp-node-01 订阅 ocpp:cluster:node:ocpp-node-01
ocpp-node-02 订阅 ocpp:cluster:node:ocpp-node-02
```

来源节点要转发给 `ocpp-node-02` 时，会执行：

```text
PUBLISH ocpp:cluster:node:ocpp-node-02 {json}
```

Redis Pub/Sub 不持久化消息。目标节点离线、Redis 连接中断或订阅尚未建立时，消息不会补投。

### 5.3 全局限流 Key

启用 Redis 全局限流后，每秒使用一个计数 key：

```text
ocpp:inbound:qps:{epochSecond}
```

TTL 为 2 秒，用于在多实例间统计当前秒入站消息总量。

## 6. 跨节点下发链路

假设充电桩 `CP001` 连接在节点 B，业务请求打到节点 A。

```text
节点 A OcppTemplate.call("CP001", ...)
  -> 本地 sessionRepository.get("CP001") 未命中
  -> Redis lookup ocpp:session:cp:CP001 得到 nodeId = 节点 B
  -> RedisOcppClusterForwarder.forward(...)
  -> publish request 到 ocpp:cluster:node:{节点 B}

节点 B RedisOcppClusterForwarder.onMessage(...)
  -> handleRequest(...)
  -> localDispatcher.dispatch(...)
  -> OcppTemplate.callLocal(...)
  -> 本地找到 CP001 的 WebSocket 连接
  -> connection.send(...) 发送 OCPP CALL
  -> 收到 CALLRESULT/CALLERROR 后完成本地 Future
  -> publishResponse(...) 发回 ocpp:cluster:node:{节点 A}

节点 A RedisOcppClusterForwarder.onMessage(...)
  -> handleResponse(...)
  -> 按 requestId 找到 responseWaiters 中的 Future
  -> 完成业务侧最初拿到的 CompletableFuture
```

核心实现位置：

- 会话注册：[RedisBackedOcppSessionRepository.java](../ocpp-spring-boot-starter/src/main/java/com/charging/ocpp/starter/session/RedisBackedOcppSessionRepository.java)
- 下发入口：[OcppTemplate.java](../ocpp-spring-boot-starter/src/main/java/com/charging/ocpp/starter/service/OcppTemplate.java)
- Redis 转发与订阅：[RedisOcppClusterForwarder.java](../ocpp-spring-boot-starter/src/main/java/com/charging/ocpp/starter/service/RedisOcppClusterForwarder.java)
- WebSocket 接入：[OcppWebSocketHandler.java](../ocpp-spring-boot-starter/src/main/java/com/charging/ocpp/starter/websocket/OcppWebSocketHandler.java)

## 7. 负载均衡建议

### 7.1 WebSocket 长连接

负载均衡器需要支持 WebSocket 升级，并保持长连接。Nginx 示例：

```nginx
map $http_upgrade $connection_upgrade {
    default upgrade;
    '' close;
}

upstream ocpp_backend {
    server 10.0.0.11:8080;
    server 10.0.0.12:8080;
}

server {
    listen 80;

    location /ocpp/ {
        proxy_pass http://ocpp_backend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection $connection_upgrade;
        proxy_set_header Host $host;
        proxy_read_timeout 3600s;
        proxy_send_timeout 3600s;
    }
}
```

是否需要 sticky session 取决于业务入口：

- 对充电桩 WebSocket 长连接来说，连接建立后天然固定在某个后端节点。
- 对平台侧 HTTP 下发接口来说，如果请求可能打到任意节点，就需要启用 Redis 会话注册和跨节点转发。
- 如果平台侧下发接口由上层网关按 `chargePointId` 精确路由到连接所在节点，则可以不开启跨节点转发，但需要额外维护路由逻辑。

### 7.2 超时配置

建议：

- 负载均衡 WebSocket 空闲超时大于充电桩心跳周期。
- `ocpp.connection-timeout-seconds` 大于正常充电桩响应耗时。
- `ocpp.redis-session-registry-ttl-seconds` 设置为心跳周期的 2 到 3 倍。
- Redis 客户端 `timeout` 不宜过大，避免 Redis 故障时阻塞业务线程过久。

## 8. 启动步骤

1. 启动 Redis。
2. 确认业务服务可以连接 Redis。
3. 为每个服务实例设置唯一 `ocpp.node-id`。
4. 启动多个业务服务实例。
5. 通过负载均衡器或直接地址连接充电桩。
6. 调用业务接口触发平台侧下发命令。

本地双节点测试可以使用两个端口和两个 `node-id`：

```bash
java -jar ocpp-demo-business.jar --server.port=8081 --ocpp.node-id=local-node-01
java -jar ocpp-demo-business.jar --server.port=8082 --ocpp.node-id=local-node-02
```

## 9. 验证方法

### 9.1 验证会话注册

充电桩连接后执行：

```bash
redis-cli GET ocpp:session:cp:CP001
redis-cli TTL ocpp:session:cp:CP001
```

预期：

- `GET` 返回某个 `nodeId`。
- `TTL` 返回正数。

断开连接后：

```bash
redis-cli GET ocpp:session:cp:CP001
```

预期 key 被删除；如果节点异常退出，则等待 TTL 过期。

### 9.2 验证节点订阅

可以临时监听 Redis channel：

```bash
redis-cli PSUBSCRIBE ocpp:cluster:node:*
```

触发跨节点下发后，应看到 `kind=request` 和 `kind=response` 消息。

### 9.3 验证跨节点下发

1. 让 `CP001` 连接到节点 B。
2. 在 Redis 确认 `ocpp:session:cp:CP001` 的值是节点 B 的 `nodeId`。
3. 将平台下发请求发送到节点 A。
4. 节点 A 本地查不到连接，会 publish 到节点 B。
5. 节点 B 通过本地 WebSocket 发给 `CP001`。
6. 节点 A 最终收到 `CompletableFuture` 结果。

### 9.4 验证限流

启用：

```yaml
ocpp:
  max-inbound-messages-per-second: 100
  redis-global-guard-enabled: true
```

观察 Redis：

```bash
redis-cli keys 'ocpp:inbound:qps:*'
```

入站消息超过阈值时，`OcppWebSocketHandler` 会快速返回错误，避免继续占用入站处理线程池。

## 10. 监控建议

建议监控：

- WebSocket 当前连接数。
- 入站线程池活跃线程数、队列长度、拒绝次数。
- PendingRequest 数量和超时数量。
- Redis QPS、延迟、连接数、慢查询。
- `ocpp:session:cp:*` key 数量。
- 跨节点 request/response 失败日志。
- JVM CPU、堆内存、GC 暂停。

当前代码未内置 Micrometer 指标，可以在业务工程中包装 `OcppGateway`、`OcppSessionRepository` 或日志采集侧补充指标。

## 11. 常见问题

### 11.1 平台下发提示充电桩不在线

检查：

- 充电桩是否已经建立 WebSocket 连接。
- `chargePointId` 是否和连接 URL 中一致。
- Redis 中是否存在 `ocpp:session:cp:{chargePointId}`。
- `cross-node-forward-enabled` 和 `redis-session-registry-enabled` 是否同时为 `true`。
- 目标节点是否存活并订阅 `ocpp:cluster:node:{nodeId}`。

### 11.2 多个节点都收到同一条转发消息

通常是多个实例配置了相同 `ocpp.node-id`。需要保证每个实例唯一。

### 11.3 Redis 中归属节点存在，但目标节点本地找不到连接

可能原因：

- 连接刚断开，Redis key 尚未删除。
- 节点异常退出，等待 TTL 过期。
- 同一桩重连过程中存在短暂竞态。
- `redis-session-registry-ttl-seconds` 设置过长，脏路由保留时间过久。

目标节点本地找不到连接时，`callLocal(...)` 会返回充电桩不在线异常。

### 11.4 跨节点请求无响应

检查：

- Redis Pub/Sub 是否可用。
- 目标节点是否在线。
- `ocpp.connection-timeout-seconds` 是否过短。
- 充电桩是否返回了 CALLRESULT 或 CALLERROR。
- 来源节点是否重启，导致 `responseWaiters` 中的 `requestId` 丢失。

### 11.5 Redis 故障时的行为

- 本地已连接的桩仍可在当前节点本地直发。
- 其它节点无法通过 Redis 查到正确归属或完成 Pub/Sub 转发。
- Redis 全局限流不可用时，限流行为取决于 Redis 调用是否成功；生产环境应监控 Redis 可用性并设置合理超时。

## 12. 生产参数建议

| 参数 | 建议 |
| --- | --- |
| `ocpp.node-id` | 使用 Pod 名、实例 ID 或主机名，保证全局唯一 |
| `ocpp.redis-session-registry-ttl-seconds` | 心跳周期 2 到 3 倍 |
| `ocpp.connection-timeout-seconds` | 大于正常桩响应时间，常见 30 到 60 秒 |
| `ocpp.inbound-worker-threads` | 从 CPU 核数 2 到 4 倍开始压测 |
| `ocpp.inbound-queue-capacity` | 按峰值流量和可接受排队时间设置 |
| `ocpp.max-inbound-messages-per-second` | 结合单节点或集群容量压测确定 |
| `spring.redis.timeout` | 建议 1 到 3 秒，避免 Redis 故障拖慢主链路 |

## 13. 上线检查清单

- 每个实例 `ocpp.node-id` 唯一。
- Redis 高可用、监控和告警已配置。
- 负载均衡支持 WebSocket upgrade。
- WebSocket 空闲超时大于桩心跳周期。
- `redis-session-registry-enabled` 和 `cross-node-forward-enabled` 按部署模式正确开启。
- 压测覆盖充电桩上报、平台下发、本地直发、跨节点转发和 Redis 故障场景。
- 日志级别在生产环境不过度输出原始报文和敏感字段。
