# OCPP Protocol Framework 生产化使用指南

本文档面向准备把框架接入生产 CSMS 的团队，说明必须完成的配置、扩展点和运行检查。

## 1. 生产定位

框架负责 OCPP-J over WebSocket 的协议通道能力，包括：

- WebSocket 端点注册与子协议协商；
- OCPP `CALL` / `CALLRESULT` / `CALLERROR` 编解码；
- 基于 `OcppVersion + Action` 的业务 Handler 路由；
- 官方 JSON Schema 校验；
- OCPP 1.6 / 2.0.1 全 Action Request/Response 模型映射；
- 平台主动下发命令的 pending request 管理，包括连接关闭、重复连接替换旧会话和容器关闭时的请求取消；
- 默认的握手 Token 校验、Origin 白名单和报文大小限制。

业务项目仍然需要实现设备、站点、订单、计费、账户、电价、交易状态机、告警、证书和厂商兼容策略。

## 2. 推荐配置

```yaml
ocpp:
  path: /ocpp/{chargePointId}
  supported-sub-protocols:
    - ocpp1.6
    - ocpp2.0.1
  connection-timeout-seconds: 60
  allow-unknown-actions: false
  enable-default-handlers: false
  duplicate-connection-policy: CLOSE_OLD
  max-text-message-bytes: 262144
  production-readiness-check: true
  fail-on-unsafe-production-config: true
  allowed-origins:
    - "https://csms.example.com"
  require-auth-token: true
  auth-token-header: X-OCPP-Token
  auth-token-query-parameter: token
  charge-point-tokens:
    CP001: "replace-with-strong-random-token"
```

生产环境请特别注意：

1. `allowed-origins` 不要使用默认的 `*`。
2. `require-auth-token` 建议开启，并给每个 `chargePointId` 配置独立强随机 Token。
3. `allow-unknown-actions` 应保持 `false`，避免未知协议动作被静默吞掉。
4. `enable-default-handlers` 应保持 `false`，业务项目必须显式实现自己的 Handler。
5. `duplicate-connection-policy` 建议使用 `CLOSE_OLD` 或 `REJECT_NEW`，避免同一 `chargePointId` 多连接导致主动下发路由不确定。
6. `max-text-message-bytes` 应结合真实桩厂报文大小、代理限制和内存容量设置。
7. `production-readiness-check` 建议开启；上线前建议将 `fail-on-unsafe-production-config` 设为 `true`，让高风险配置在启动阶段失败。

## 3. 自定义业务 Handler

```java
@Component
public class ProductionOcpp16Handler {
    @OcppActionMapping(version = OcppVersion.OCPP_16, action = "BootNotification")
    public BootNotificationResponse boot(OcppRequestContext context, BootNotificationRequest request) {
        // 1. 校验 chargePointId 是否已在业务系统注册
        // 2. 保存厂商、型号、固件版本等设备信息
        // 3. 返回 Accepted / Pending / Rejected
        BootNotificationResponse response = new BootNotificationResponse();
        response.setStatus("Accepted");
        response.setCurrentTime(Instant.now().toString());
        response.setInterval(300);
        return response;
    }
}
```

## 4. 主动下发命令

优先使用 `OcppGateway` 的便捷方法，减少业务代码手写 Action 字符串：

```java
@RestController
class ChargePointCommandApi {
    private final OcppGateway ocppGateway;

    ChargePointCommandApi(OcppGateway ocppGateway) {
        this.ocppGateway = ocppGateway;
    }

    @PostMapping("/charge-points/{id}/remote-start")
    CompletableFuture<RemoteStartTransactionResponse> remoteStart(@PathVariable String id,
                                                                   @RequestBody RemoteStartTransactionRequest request) {
        return ocppGateway.remoteStartTransaction16(id, request);
    }
}
```

如果框架暂未提供某个 Action 的便捷方法，可以使用通用 `call`：

```java
SetVariablesRequest payload = SetVariablesRequest.builder()
        .setVariableData(Collections.singletonList(
                SetVariableData.builder()
                        .component(Component.builder().name("TxCtrlr").build())
                        .variable(Variable.builder().name("Enabled").build())
                        .attributeValue("true")
                        .build()))
        .build();

CompletableFuture<SetVariablesResponse> future =
        ocppGateway.call("CP001", OcppVersion.OCPP_201, "SetVariables", payload, SetVariablesResponse.class);
```

当充电桩连接关闭、同一 `chargePointId` 以 `CLOSE_OLD` 策略重复连接，或 Spring 容器关闭时，该会话上等待中的主动下发请求会立即以异常完成。业务代码应统一处理 `CompletableFuture` 的异常分支，避免把断线误判为业务成功。

## 5. 必须替换的生产扩展点

### 5.1 会话仓储

默认 `InMemoryOcppSessionRepository` 仅适合单机开发。多实例部署时必须提供自定义 `OcppSessionRepository`，并结合消息总线或网关实现跨节点命令路由。

```java
@Bean
public OcppSessionRepository ocppSessionRepository() {
    return new RedisBackedOcppSessionRepository(...);
}
```

### 5.2 协议状态机

DTO 与 JSON Schema 只能保证报文字段结构合法，不能保证业务状态正确。生产系统应额外维护：

- BootNotification 状态；
- Heartbeat 最近活跃时间；
- Connector / EVSE 状态；
- StartTransaction、StopTransaction、TransactionEvent 幂等；
- MeterValues 补传和时间漂移处理；
- OCPP 2.0.1 Device Model 变量状态。

### 5.3 观测与审计

建议在业务 Handler 或 WebSocket 层外围接入：

- 连接数、在线桩数、断连次数；
- Action QPS、耗时、失败率；
- CALL 超时数量；
- CALLERROR 错误码分布；
- 必要的报文审计与敏感字段脱敏。
