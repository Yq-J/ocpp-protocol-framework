# OCPP 协议框架开发使用手册

## 1. 框架目标

本框架基于 Spring Boot、JDK8、Maven、Lombok 和 Spring WebSocket 实现，目标是将 OCPP 协议实现与业务系统彻底解耦。业务开发者不需要直接处理 WebSocketSession、OCPP-J 数组帧、CALL/CALLRESULT/CALLERROR、uniqueId、请求响应关联、超时清理、动作路由和异常转换。

业务侧只需要关注两类入口：

1. 充电桩主动上报时，使用 `@OcppActionMapping` 编写业务处理方法。
2. 中心系统主动下发命令时，注入 `OcppGateway` 或 `OcppTemplate` 调用 `call` 方法。

## 2. 工程结构

```text
ocpp-protocol-framework-v2
├── ocpp-core
│   ├── enums              # OCPP 版本、消息类型
│   ├── exception          # OCPP 错误码和协议异常
│   ├── protocol           # OCPP-J 帧模型和编解码器
│   ├── handler            # 动作处理器、注册表、请求上下文
│   ├── session            # 会话仓储和连接抽象
│   ├── schema             # Schema 校验扩展点
│   └── model              # OCPP 1.6 / 2.0.1 常用 DTO
├── ocpp-spring-boot-starter
│   ├── autoconfigure      # Spring Boot 自动装配
│   ├── registry           # 注解方法扫描与反射调用
│   ├── service            # OcppTemplate 主动下发命令、Redis 转发、限流
│   ├── session            # Redis 会话注册仓储
│   └── websocket          # 基于 spring-websocket 的传输层
└── ocpp-demo-business     # 业务接入示例
```

## 3. 技术选型

- JDK：1.8。
- Spring Boot：2.7.x。
- WebSocket：显式使用 `org.springframework:spring-websocket`。
- JSON：Jackson。
- 注解简化：Lombok。
- 构建方式：Maven 多模块工程。

本框架没有使用 Netty、STOMP 或 SockJS。OCPP JSON over WebSocket 的通信形态是原生 WebSocket 文本帧，因此 starter 中使用 `TextWebSocketHandler` 和 `WebSocketConfigurer`。

## 4. IDEA 导入

1. 解压 `ocpp-protocol-framework-v2.zip`。
2. 使用 IDEA 打开根目录。
3. 等待 Maven 自动导入三个模块。
4. 安装 IDEA Lombok 插件。
5. 开启 Annotation Processing。
6. 运行 `ocpp-demo-business` 模块中的 `OcppDemoApplication`。

## 5. 构建、安装与依赖引入

在根目录执行：

```bash
mvn clean install
```

业务系统引入 starter：

```xml
<dependency>
    <groupId>com.charging.ocpp</groupId>
    <artifactId>ocpp-spring-boot-starter</artifactId>
    <version>1.0.0</version>
</dependency>
```

如果业务系统也使用 Lombok，建议同时安装 IDE 插件并开启注解处理。

## 6. 配置项

示例配置：

```yaml
server:
  port: 8080

ocpp:
  path: /ocpp/{chargePointId}
  connection-timeout-seconds: 60
  allow-unknown-actions: false
  inbound-worker-threads: 32
  inbound-queue-capacity: 300000
  pending-cleanup-interval-millis: 1000
  high-load-log-sampling-enabled: true
  max-inbound-messages-per-second: 12000

  redis-global-guard-enabled: true
  redis-session-registry-enabled: true
  cross-node-forward-enabled: true
  redis-session-registry-ttl-seconds: 180
  node-id: ${HOSTNAME:ocpp-node-01}
```

默认 WebSocket 地址：

```text
ws://localhost:8080/ocpp/{chargePointId}
```

充电桩应在握手阶段携带子协议：

- OCPP 1.6：`ocpp1.6`
- OCPP 2.0.1：`ocpp2.0.1`

配置项说明：

| 配置项 | 默认值 | 说明 |
| --- | --- | --- |
| `ocpp.path` | `/ocpp/{chargePointId}` | WebSocket 接入路径，`chargePointId` 会在握手阶段提取。 |
| `ocpp.connection-timeout-seconds` | `60` | 平台下发 CALL 后等待 CALLRESULT/CALLERROR 的超时时间。 |
| `ocpp.allow-unknown-actions` | `false` | 收到未注册 Action 时是否自动返回空对象。 |
| `ocpp.inbound-worker-threads` | CPU 核数相关 | 入站消息业务处理线程数。 |
| `ocpp.inbound-queue-capacity` | `200000` | 入站消息处理队列容量。 |
| `ocpp.pending-cleanup-interval-millis` | `1000` | PendingRequest 超时清理扫描周期。 |
| `ocpp.high-load-log-sampling-enabled` | `true` | 高负载时是否降低部分日志输出。 |
| `ocpp.max-inbound-messages-per-second` | `0` | 入站消息每秒上限，`<=0` 表示不限流。 |
| `ocpp.redis-global-guard-enabled` | `false` | 是否使用 Redis 做多实例全局入站限流。 |
| `ocpp.redis-session-registry-enabled` | `false` | 是否将 `chargePointId -> nodeId` 写入 Redis。 |
| `ocpp.cross-node-forward-enabled` | `false` | 是否启用 Redis Pub/Sub 跨节点下发转发。 |
| `ocpp.redis-session-registry-ttl-seconds` | `120` | Redis 会话归属 key 的 TTL。 |
| `ocpp.node-id` | JVM 名称 | 当前实例节点标识，多节点必须全局唯一。 |

`OcppWebSocketHandshakeHandler` 当前固定支持 `ocpp1.6` 和 `ocpp2.0.1`。如果要支持新的子协议，需要修改握手处理器和 `OcppVersion` 映射。

## 7. 业务层处理充电桩上报

业务处理器示例：

```java
@Component
@Slf4j
public class DemoOcppBusinessHandler {

    @OcppActionMapping(version = OcppVersion.OCPP_16, action = "BootNotification")
    public BootNotificationResponse onBoot16(OcppRequestContext context,
                                             BootNotificationRequest request) {
        log.info("收到启动通知，桩号={}，厂商={}",
                context.getChargePointId(), request.getChargePointVendor());

        BootNotificationResponse response = new BootNotificationResponse();
        response.setCurrentTime("2026-05-21T00:00:00Z");
        response.setInterval(300);
        response.setStatus("Accepted");
        return response;
    }
}
```

说明：

- `OcppRequestContext` 提供 chargePointId、sessionId、version、action、uniqueId 等上下文。
- 请求 DTO 由框架自动从 JSON payload 转换。
- 返回 DTO 由框架自动封装为 OCPP CALLRESULT。
- 抛出的协议异常或业务异常会被转换为 CALLERROR。

## 8. 业务层向充电桩下发命令

推荐依赖接口：

```java
@Autowired
private OcppGateway ocppGateway;
```

下发 OCPP 1.6 远程启动：

```java
RemoteStartTransactionRequest request = new RemoteStartTransactionRequest();
request.setConnectorId(1);
request.setIdTag("USER001");

CompletableFuture<RemoteStartTransactionResponse> future = ocppGateway.call(
        "CP001",
        OcppVersion.OCPP_16,
        "RemoteStartTransaction",
        request,
        RemoteStartTransactionResponse.class
);
```

框架会自动完成：

- 生成 uniqueId。
- 编码为 `[2, uniqueId, action, payload]`。
- 优先查找本节点在线连接。
- 本节点无连接时，可按 Redis 会话归属索引跨节点转发。
- 通过 Spring WebSocket 发送文本帧。
- 保存 PendingRequest。
- 收到 CALLRESULT 时完成 Future。
- 收到 CALLERROR 时完成异常。
- 超时后清理等待中的请求。

主动下发的本地链路：

```text
业务代码
 -> OcppGateway.call(...)
 -> OcppTemplate.call(...)
 -> sessionRepository.get(chargePointId)
 -> OcppTemplate.callLocal(...)
 -> OcppCodec.encodeCall(...)
 -> OcppConnection.send(...)
 -> 充电桩
 -> CALLRESULT/CALLERROR
 -> OcppWebSocketHandler
 -> OcppTemplate.completeResult / completeError
 -> CompletableFuture 完成
```

主动下发的跨节点链路：

```text
业务代码请求到节点 A
 -> 节点 A 本地无 CP001 连接
 -> 节点 A 查询 Redis：ocpp:session:cp:CP001
 -> 得到 targetNodeId = 节点 B
 -> 节点 A publish request 到 ocpp:cluster:node:{节点 B}
 -> 节点 B 收到 request
 -> 节点 B 执行 OcppTemplate.callLocal(...)
 -> 节点 B 使用本地 WebSocket 发送给 CP001
 -> 节点 B 收到桩响应后 publish response 到 ocpp:cluster:node:{节点 A}
 -> 节点 A handleResponse(...)
 -> 业务侧 CompletableFuture 完成
```

## 9. Lombok 使用说明

本次增强已在 DTO 和关键类中使用常见 Lombok 注解：

| 注解 | 用途 |
| --- | --- |
| `@Data` | 自动生成 getter、setter、toString、equals、hashCode。 |
| `@Builder` | 为 DTO 提供链式构建方式。 |
| `@NoArgsConstructor` | 生成无参构造器，便于 Jackson 反序列化。 |
| `@AllArgsConstructor` | 生成全参构造器，便于测试和构造对象。 |
| `@Accessors(chain = true)` | setter 返回 this，支持链式赋值。 |
| `@Slf4j` | 自动生成日志对象，替代手写 LoggerFactory。 |
| `@RequiredArgsConstructor` | 为 final 字段生成构造器，适合 Spring 构造器注入。 |

## 10. WebSocket 实现说明

starter 模块显式依赖 `spring-websocket`。核心类如下：

- `OcppWebSocketConfigurer`：注册 WebSocket 端点。
- `OcppHandshakeInterceptor`：从路径中解析 chargePointId。
- `OcppWebSocketHandshakeHandler`：处理子协议协商。
- `OcppWebSocketHandler`：处理连接、文本帧、路由、响应、异常转换。
- `SpringOcppConnection`：封装 Spring WebSocketSession，向 core 暴露统一连接接口。

协议层不直接依赖业务代码；业务层也不直接依赖 WebSocketSession。

## 11. OCPP 1.6 和 2.0.1 兼容方式

框架通过 `OcppVersion` 区分协议版本，通过 `version + action` 路由到不同业务处理器。这样即使两个版本存在同名 Action，也不会路由冲突。

内置 DTO 覆盖常见动作，例如：

- OCPP 1.6：BootNotification、Authorize、Heartbeat、StartTransaction、StopTransaction、StatusNotification、MeterValues、RemoteStartTransaction、RemoteStopTransaction。
- OCPP 2.0.1：BootNotification、Authorize、Heartbeat、StatusNotification、MeterValues、TransactionEvent、RequestStartTransaction、RequestStopTransaction。

## 12. 如何扩展更高版本 OCPP

以扩展 OCPP 2.1 为例：

1. 在 `OcppVersion` 中新增版本枚举和值到子协议的映射。
2. 在 `model` 下新增 `v21` DTO 包。
3. 增加 OCPP 2.1 的请求与响应 DTO。
4. 在业务层新增 `@OcppActionMapping(version = OcppVersion.OCPP_21, action = "...")`。
5. 如需要强 Schema 校验，实现 `OcppSchemaValidator`，替换默认 `NoopOcppSchemaValidator`。
6. 如需要版本级默认处理器，可在 starter 中新增 `DefaultOcpp21Handlers`。
7. 如 OCPP 2.1 引入新的 WebSocket 子协议名，在配置和握手处理逻辑中补充。

核心设计点是：编解码器只关心 OCPP-J 的公共帧结构；版本差异主要体现在 DTO、Action 和业务处理器中。

## 13. Redis 会话注册与跨节点转发

### 13.1 为什么需要 Redis 会话注册

默认 `InMemoryOcppSessionRepository` 适合单节点或本地开发。它只在当前 JVM 内保存：

```text
chargePointId -> OcppConnection
sessionId     -> chargePointId
```

多节点部署后，充电桩 WebSocket 连接可能在节点 B，但平台侧 HTTP 下发请求可能打到节点 A。节点 A 的本地内存没有该充电桩连接，如果不做路由，就只能返回“充电桩不在线”。

启用 `redis-session-registry-enabled=true` 后，框架会额外在 Redis 写入：

```text
ocpp:session:cp:{chargePointId} -> {nodeId}
```

这条记录只表示连接归属，不表示 Redis 保存了连接对象。真正的 WebSocket 连接仍然在目标节点 JVM 内。

### 13.2 连接建立和断开时发生什么

连接建立时：

```text
OcppWebSocketHandler.afterConnectionEstablished(...)
 -> readChargePointId(session)
 -> OcppVersion.fromSubProtocol(...)
 -> sessionRepository.save(new SpringOcppConnection(...))
```

如果当前仓储是 `RedisBackedOcppSessionRepository`，`save()` 会：

1. 写入本地 `localByChargePointId`，供当前节点直发。
2. 写入本地 `localSessionToChargePointId`，供断链清理。
3. 写入 Redis key `ocpp:session:cp:{chargePointId}`，value 为当前 `nodeId`。

连接断开时：

```text
OcppWebSocketHandler.afterConnectionClosed(...)
 -> sessionRepository.remove(session.getId())
```

`remove()` 会先删除本地索引，再检查 Redis 中的 value 是否仍等于当前节点 `nodeId`。只有仍归属于当前节点时才删除 Redis key，避免旧连接断开时误删新节点写入的归属关系。

### 13.3 跨节点转发如何生效

跨节点转发需要同时满足：

- `ocpp.redis-session-registry-enabled=true`
- `ocpp.cross-node-forward-enabled=true`
- Redis 连接可用，Spring 容器中存在 `StringRedisTemplate` 和 `RedisMessageListenerContainer`
- 当前 `OcppSessionRepository` 是 `RedisBackedOcppSessionRepository`
- `ocpp.node-id` 在所有实例中唯一

每个节点启动时都会订阅自己的私有 channel：

```text
ocpp:cluster:node:{nodeId}
```

例如：

```text
nodeId=ocpp-node-01 订阅 ocpp:cluster:node:ocpp-node-01
nodeId=ocpp-node-02 订阅 ocpp:cluster:node:ocpp-node-02
```

节点 A 下发给 `CP001` 时，如果本地没有连接，会读取：

```text
GET ocpp:session:cp:CP001
```

如果返回 `ocpp-node-02`，节点 A 会 publish 到：

```text
ocpp:cluster:node:ocpp-node-02
```

节点 B 收到 `kind=request` 后，调用本地 `OcppTemplate.callLocal(...)`，通过本地 WebSocket 发给充电桩。执行完成后，节点 B 再 publish `kind=response` 回节点 A 的私有 channel。

### 13.4 Redis Pub/Sub 消息结构

跨节点 request 包含：

```json
{
  "kind": "request",
  "requestId": "uuid",
  "sourceNodeId": "ocpp-node-01",
  "chargePointId": "CP001",
  "version": "OCPP_16",
  "action": "RemoteStartTransaction",
  "payload": {}
}
```

跨节点 response 包含：

```json
{
  "kind": "response",
  "requestId": "uuid",
  "ok": true,
  "payload": {}
}
```

失败时：

```json
{
  "kind": "response",
  "requestId": "uuid",
  "ok": false,
  "error": "充电桩不在线：CP001"
}
```

`requestId` 只用于节点间转发响应关联；OCPP 协议帧内部仍然使用 `OcppTemplate.callLocal(...)` 生成的 OCPP `uniqueId`。

### 13.5 关键代码位置

| 能力 | 类 |
| --- | --- |
| Redis 会话注册 | `RedisBackedOcppSessionRepository` |
| 平台下发入口 | `OcppTemplate.call(...)` |
| 本地 WebSocket 下发 | `OcppTemplate.callLocal(...)` |
| Redis Pub/Sub 转发 | `RedisOcppClusterForwarder.forward(...)` |
| Redis 消息订阅处理 | `RedisOcppClusterForwarder.onMessage(...)` |
| 收到 request 后本地执行 | `RedisOcppClusterForwarder.handleRequest(...)` |
| 回传 response | `RedisOcppClusterForwarder.publishResponse(...)` |
| 来源节点完成 Future | `RedisOcppClusterForwarder.handleResponse(...)` |

### 13.6 调试验证

连接充电桩后查看归属：

```bash
redis-cli GET ocpp:session:cp:CP001
redis-cli TTL ocpp:session:cp:CP001
```

监听跨节点消息：

```bash
redis-cli PSUBSCRIBE ocpp:cluster:node:*
```

触发平台下发后，应看到一条 `kind=request` 和一条 `kind=response`。

### 13.7 使用边界

- Redis Pub/Sub 不持久化消息，目标节点离线时消息会丢。
- `node-id` 不唯一会导致多个实例订阅同一个 channel，从而重复处理同一条转发消息。
- Redis 会话注册 TTL 过长会延长脏路由存在时间，过短可能在异常情况下提前过期。
- PendingRequest 仍然是节点内内存管理，不建议跨节点共享 WebSocketSession 或 PendingRequest。
- 目标节点收到 request 后仍会校验本地连接是否存在；如果本地连接已断开，会返回异常。

## 14. 安全建议

公网接入充电桩时建议：

- 使用 WSS。
- 在握手阶段校验桩编号、签名、token 或证书。
- 限制 allowed origins。
- 对原始报文做审计留存。
- 对敏感日志做脱敏。
